# 入戏 v2.0 —— 总纲（架构升级：存储 + 双模式内容架构）

> 本文档定义 v2.0 大版本的范围边界、核心架构决策、数据模型总纲，以及内部小版本的粗分步骤。不含任何单个小版本的详细实施指令——detail级文档会针对每个小版本单独产出。

---

## 一、v2.0 要解决的核心问题

1. **持久化风险**：当前dev模式对剧本/prompt的在线编辑，在Render重启/重新部署后会丢失，唯一的持久化手段是手动导出后提交回git。这是数据安全问题，必须优先解决。
2. **内容形态单一**：当前只支持"单局、限定轮数"一种玩法结构，无法承载多章节、长篇幅、带分支剧情的原创互动小说内容。
3. **架构前瞻性不足**：v1.x的剧本schema和engine设计没有为多章节、跨局记忆、剧情分支预留空间，如果不先做数据模型层面的统一设计就直接叠加长篇功能，会导致两套玩法在代码和内容结构上"缝合"，难以维护。

---

## 二、v2.0 范围定义

### 做什么

- 存储层从"本地文件系统+手动导出"迁移到 **Supabase**（Postgres + Storage），实现内容资产（剧本/prompt/echo文案）的生产级持久化
- 引入统一的**"作品(work)"数据模型**，覆盖短局与长篇两种内容形态，替代v1.x中"剧本(script)"这一单一概念
- 设计并落地**章节剧情树**机制：多章节、flag系统（跨章节记忆）、章节收尾摘要、硬性分支+轻度AI自由选择两种出口类型
- 落地长篇玩法的**存档/读档**机制
- 产品层面显性区分并包装两种玩法模式：**「特写」**（短局，原v1.0-v1.2玩法的产品化命名）与**「长镜」**（长篇，本次新增）
- 选择页、卡片视觉、后端engine/router做相应适配，支持两种模式共存

### 不做什么（明确排除，留给v2.1+）

- **重度AI自由**：AI在运行时创造全新分支/新章节内容（本次只做"轻度自由"——AI在预先写好的候选章节池中选择走向）
- **多模态**（立绘、场景插画、BGM等）——数据模型预留字段，但v2.0不实际接入
- **Flag的删改机制**——v2.0先支持flag只增，删改能力作为独立迭代排入v2.1之后（见下方架构决策注记）
- **前端框架整体升级**（Vue3/SvelteKit等）——v2.0仍基于v1.3已确定的 Tailwind + Alpine.js，是否需要升级视v2.0落地后的实际复杂度再评估，不在本次范围内预判
- **难度分级功能**——如v1.x尚未实现，仍按原计划独立排期，不并入v2.0
- **Echo卡片文案的LLM动态生成**（v1.3已定为v1.4+事项，v2.0维持模板池方案不变）

## 三、核心架构决策

### 3.1 存储迁移

- 数据库：Supabase（Postgres）承载作品/章节/flag定义/存档等结构化内容
- 文件存储：Supabase Storage（或后续视需要引入Cloudflare R2）承载未来多模态资源，本次仅预留，不实际使用
- 代码资产（Python/JS源码）继续走 git；**内容资产**（作品/章节/prompt/存档）全部迁移到数据库，两类资产的持久化路径彻底分离
- dev模式保存操作直接写库，不再依赖服务运行时的文件系统，重启/重新部署不再导致内容丢失
- v1.x已有剧本需要全量迁移到新的"work"数据模型（见3.2），迁移后原剧本行为应与迁移前完全一致

### 3.2 短局与长篇：统一为单一"作品(work)"类型

**决策**：工程与数据层面不做两套并行结构，"短局"与"长篇"统一抽象为同一个 `work` 类型，短局是长篇的退化形态：

- 短局作品：`type: "short_form"`，仅含1个章节节点，该章节的所有出口直接判定为结局，不涉及flag系统的跨章节引用、不涉及多章节分支树
- 长篇作品：`type: "long_form"`，包含多个章节节点，通过flag系统和章节出口机制形成分支剧情树

**为什么这样做**：
- 选择页、后端engine、router层可以共用同一套数据结构和大部分逻辑，不需要维护两套并行代码
- v1.x已有剧本理论上只需要包一层"单章节work"封装即可完成迁移，无需重写内容本身
- 两种模式的差异被收敛为"章节数量"和"是否使用flag/分支机制"这两个维度上的差异，而非底层结构差异

**唯一的产品层面区分**：短局与长篇在**产品命名、入口UI、卡片视觉**上必须清晰区分（见第五节），工程统一不代表产品体验混同。

### 3.3 数值系统：作品级贯穿

- 数值维度（如 a/b/c/d/e）在作品级定义一次（含 initial/min/max），全部维度对所有章节**全程可见、全程参与分支判定**
- 每章节**不**通过额外配置声明"激活哪些维度"，而是**由该章节的 key_points/pitfalls 涉及到的维度集合自动推导**出这一章"实际会变动"的维度范围
- 约束规则：导演agent的自由裁量部分（不对应任何key_point的基础态度反馈增量），**只允许在本章key_points/pitfalls已涉及的维度集合内**给分，不允许触碰集合之外的维度，避免跨维度的意外漂移
- 好处：零额外配置成本，数值体验连贯不割裂，同时保留最小护栏防止LLM自由裁量越界

### 3.4 长篇模式的"记忆"机制：Flag + 章节摘要 混合

- **Flag（结构化，驱动分支判定）**：跨章节持续存在的 `{flag_name: value}` 字典，记录"发生过的关键剧情事实"，用于硬性分支条件判定，纯代码比较，可复现、可测试
- **章节摘要（细节呼应）**：每章收尾时由导演agent生成一句轻量摘要（30-50字量级），存入存档，供后续章节prompt引用，弥补flag无法承载语气/细节层面记忆的局限
- 分工明确：**flag决定"剧情走向"，摘要只影响"呈现质感"**，绝不用摘要或LLM自由判断来决定分支走向，保持长篇模式与v1.0关键点系统一脉相承的"可控可复现"设计原则

**架构注记（v2.0范围内的已知简化）**：v2.0的flag系统仅支持写入（只增），不支持删除/修改已写入的flag。这意味着v2.0暂不支持"剧情反悔"类设计（如某个已建立的事实被后续行为撤销）。Flag的删改能力将作为**v2.1之后的独立迭代**排入路线图，届时需要重新评估flag的版本历史/覆盖规则，避免影响已依赖flag只增特性设计的早期长篇内容。

### 3.5 章节剧情树的出口机制：硬性分支 + 轻度AI自由选择

每个章节节点的"出口(exits)"支持两种类型共存：

- **硬性分支（hard_condition）**：基于flag+数值的表达式判定，复用现有condition_parser，命中即走向对应下一章节点
- **轻度AI自由选择（ai_choice）**：提供一组预先写好的候选章节池，由导演agent根据玩家本章整体表现在候选池内选择一个作为走向；必须配置 `fallback_next_chapter` 作为解析失败兜底，保证流程绝不会卡死
- 判定顺序：硬性分支优先判定，都不满足时才进入AI自由选择兜底

**明确排除**：章节剧情树的**结构本身**（有多少章节、多少条出口、候选池里有哪些章节）完全由人类剧本作者预先写定，游玩过程不会新增或改变这张图的形状。AI只负责"在人类划定的范围内选择/生成内容"，不负责"决定图长什么样"。重度自由（AI运行时创造全新分支）明确排除在v2.0之外。

## 四、数据模型总纲

以下为v2.0的核心数据结构范围（字段级详细schema与迁移脚本在v2.0.2 detail文档中给出）：

- **作品(work)**：作品级元信息（标题/标签/teaser等展示字段）+ 作品级数值维度定义 + 章节清单 + 入口章节
- **章节(chapter)**：单章节的剧本内容（沿用v1.0的background/ai_character/key_points/pitfalls结构）+ flag读写声明 + 出口(exits)定义
- **Flag字典**：存档级别的扁平 `{flag_name: value}` 结构，只增不改（v2.0范围内）
- **存档(save)**：当前章节位置 + 当前轮次 + 全局数值状态 + flag字典 + 章节摘要列表 + 当前章节内的对话历史与已命中关键点集合

## 五、产品层面：模式命名与区分

### 命名

- **「特写」**——单次游走，轮数限定。每一次抉择都是放大镜下的关键帧，一念落定，故事收束。（对应 `short_form`）
- **「长镜」**——多章连载，世界生长。故事不落幕，跟随镜头一镜到底，在漫长叙事中与角色一同呼吸。（对应 `long_form`）

### 显性区分要求

- 选择页需要在**主题标签筛选之上**，设置更高优先级的模式切换入口（如"特写/长镜"两个大按钮或tab），因为"我现在有多少时间"是玩家决策的第一层判断，应比主题标签更显眼
- 两种模式的作品卡片**共用同一套卡片骨架**（复用v1.1已有的标签徽章+标题+teaser+底部信息行结构），长篇卡片在底部信息位额外展示进度信息（如"第3/8章"或"未开始/进行中/已完结"状态角标），短局卡片保持v1.1现状
- 命名与视觉包装延续v1.3"世界残响(Echoes)"确定的文学化调性，不使用暴露内部数据结构差异的直白措辞（如"短局/长篇"这类工程术语不应出现在玩家可见的UI文案中）

## 六、v2.0 内部小版本迭代规划（粗分）

| 版本 | 内容 | 产出形式 |
|---|---|---|
| v2.0.1 | 存储迁移：Supabase落地，数据库表结构建立，v1.x已有剧本全量迁移为work数据模型，dev模式改为直接写库 | 已可独立验证：重启/重新部署后内容不丢失 |
| v2.0.2 | work/chapter/flag/存档数据模型细化落地 + 后端engine适配（含短局作为退化形态的兼容验证） | 后端支持长篇数据结构，但玩法层暂未接入前端 |
| v2.0.3 | 长篇玩法MVP：章节推进、存档/读档、硬性分支判定、ai_choice轻度自由选择机制跑通 | 可用一个最小的多章节长篇作品完整验证机制，暂不做视觉打磨 |
| v2.0.4 | 产品层选择页改版：「特写/长镜」模式切换入口、长篇卡片进度展示、章节内阅读流呈现适配 | 玩家可见的完整长篇体验上线 |
| v2.0.5 | 前端复杂度评估：视v2.0.3/2.0.4实际落地后的交互复杂度，判断是否需要升级前端框架 | 评估结论文档，非必然的框架升级实施 |

每个小版本完成后需回归验证：**既有的短局(「特写」)玩法行为不受影响**，这是贯穿整个v2.0的底线要求，因为短局是长篇的退化形态，任何数据模型或engine调整都必须先保证向下兼容。
