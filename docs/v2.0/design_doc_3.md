# 入戏 v2.0.detail-3 —— 产品层 UI 改版（特写 / 长镜）

> 本文档是 v2.0 总纲（`ruxi-v2.0-overall.md`）下的第三份、也是本大版本最后一份可执行指令文档，承接 detail-2 已落地的长篇玩法引擎。本阶段目标是让长篇（长镜）玩法在**产品体验层面**完整可用，并与短局（特写）在视觉上清晰区分、调性统一。收尾附带一次前端框架复杂度评估。

详细UI设计参考 /Users/ryoyo/Workspace/temp_workspace/intereactive_chat_script/docs/v2.0/ui_mock_example
的文件。

## 一、本阶段目标

1. 选择页新增「特写 / 长镜」模式切换入口，优先级高于v1.1已有的主题标签筛选
2. 「特写」模式卡片与现有玩家端体验**保持不变**，不做改动
3. 「长镜」模式新增专属卡片视觉：章节进度指示、结局神秘感处理（不展示固定总数）
4. 章节阅读页新增"剧情脉络线"视觉组件，延续v1.3"世界残响"整体调性，但风格上更接近有机、写意的水墨/潮汐笔触，不做成显式的决策树图
5. 收尾：评估当前 Tailwind + Alpine.js 是否足以支撑本阶段的交互复杂度，产出评估结论

## 二、本阶段明确不做

- 不改动「特写」模式的任何现有视觉与交互
- 不做多存档槽位管理UI（detail-2已限定本阶段仅单存档位，UI也不需要支持切换存档）
- 不接多模态资源展示
- 不做AI自生长分支（本阶段所有长篇内容仍是剧本作者预先写定的固定结构，UI的"神秘感"处理只是**不提前透露**，不代表底层机制变了）

## 三、命名与产品文案

- 短局对外统一称为 **「特写」**，slogan："一念落定，故事收束。"
- 长篇对外统一称为 **「长镜」**，slogan："一镜到底，与角色一同呼吸。"
- **UI文案中不得出现** `short_form`/`long_form`/`chapter`/`flag`/`exits` 等内部工程术语，所有面向玩家的文案需经过产品化包装

## 四、选择页改版

### 4.1 模式切换入口

- 位置：页面顶部，**在**v1.1已有的 `origin_tag`/`theme_tags` 筛选栏**之上**
- 形式：两个等宽分区（不用胶囊标签样式，与下方主题筛选做视觉层级区分），当前选中项用强调色 `#4B6355` 实底，未选中为透明+细边框
- 切换后，下方卡片列表、以及主题筛选栏的可选项范围（如某些主题标签可能只有短局或只有长篇作品使用），需要联动更新

### 4.2 长镜卡片新增信息位

在v1.1已有卡片骨架（标签徽章+标题+teaser+底部信息行）基础上追加：

- **章节进度指示**：一排小圆点，实心=已读章节，空心=未读章节，圆点数量对应该作品的 `chapter_ids` 长度（注意：**只在玩家已开始游玩该作品时展示**，未开始的作品不展示进度点，避免圆点数量提前暴露"这部作品有几章"这类可能影响体验的信息——具体是否展示由产品判断，若认为章节总数本身不算剧透可以调整为始终展示，本阶段先按"仅游玩中/已完结才展示"处理，更保守）
- **结局状态位**：替代原计划的"N种结局"数字展示，改为：
  - 未解锁任何结局 / 未全部解锁：展示"仍有未至之地"文案 + 月相类图标（如 `ti-moon`）
  - 已解锁该作品全部已知结局（仅对有固定终点集合的长篇作品成立，判定逻辑见第六节）：展示"已阅尽此世"文案，可选择性地此时才展示玩家已发现的结局数量
- 底部信息行左侧沿用原 `player_role_hint` 逻辑或改为"未开始"/"进行中·第N章"/"已完结"状态文案

### 4.3 短局卡片

不改动，沿用v1.1现状。

## 五、章节阅读页改版

### 5.1 剧情脉络线组件（新增，长篇专属）

- 视觉语言：**有机流动的曲线**（水墨/潮汐笔触感），不是直线连接的节点决策树图
- 已走过的路径：用连贯实线曲线绘制，途经的章节用小实心圆点标记
- 当前所在位置：用略大的空心圆环高亮
- **前路不做显式绘制**：主线在当前点之后继续延伸一小段后，通过渐变透明度自然消失（fade to nothing），不画出具体的分岔线数量，避免暴露分支结构
- **若本局曾经历过一次选择（即某个已走过的章节存在多条exits，本局只选择了其中一条）**：从对应章节节点处，画一条更细、更快消失的"支流"曲线岔开又淡出，暗示"这里曾有过别的可能"，**不标注这条支流通向哪里、有几条**
- 具体SVG实现参考本阶段配套的视觉稿（曲线路径 + 渐变透明度stroke + 独立的支流fade路径），已在前序讨论中给出示例代码结构，实现时可直接参照

### 5.2 数值条、正文排版、气泡（如适用）

沿用v1.3"世界残响"已定的设计token不变：底色 `#F5F2EB`、正文 `#232323`、边框 `#DDD8CF`、正面强调 `#4B6355`、危险 `#8A4D4A`，衬线字体（Georgia + Noto Serif SC），`line-height:1.9`，`letter-spacing:0.03em`，数值条细线轨道+缓动过渡+保留小号灰字数字，情绪标签仍为台词前的斜体舞台提示写法。长镜与特写在这一层视觉语言上**完全一致**，差异只体现在脉络线组件和章节/结局相关的信息位上。

### 5.3 章节切换过渡

- 章节结束、判定出下一章时，展示一个简短的过渡态（如渐隐+新章节标题浮现），避免生硬跳转
- 具体动效时长/形式可参考v1.3已定的 `tone_preset` 机制（从容/明快两档），本阶段默认使用"从容"档

## 六、后端配合改动

本阶段前端改动依赖以下接口层调整（如detail-2尚未覆盖，需在本阶段一并补上）：

- `GET /api/works`（或现有 `/api/scripts` 的长篇场景扩展）需要返回：`type`、`chapter_ids`长度、玩家当前存档对应的进度状态（未开始/进行中第N章/已完结）
- 新增判定逻辑：**是否已解锁全部已知结局**——遍历该work的所有章节，找出所有"终点章节"（无exits或exits标记为terminal的章节）集合作为该work的"已知结局集合"（仅对该work从未被声明为"开放式/AI自生长"的情况下才计算此集合并做判定；若该work未来标记为开放式类型，判定逻辑直接跳过，接口固定返回 `all_endings_discovered: false`）
- 存档需要记录 `discovered_endings`（玩家在该work下已到达过的终点章节id集合，跨局累积，不因重新开局而清空——具体是否跨局累积由产品决定，本阶段默认按"跨局累积"实现，即鼓励玩家多周目探索）
- 接口返回 `all_endings_discovered` 布尔值，不返回具体总数或候选终点章节列表，避免前端拿到后二次推导出分支细节

## 七、验证要求

1. 选择页模式切换正常工作，切换后卡片列表和筛选栏联动更新，「特写」现有卡片与交互不受任何影响
2. 「长镜」卡片的章节进度点、结局状态位展示正确，且未开始的作品不提前展示分支/结局相关的具体数字信息
3. 人为让测试账号走完一部demo长篇作品的**全部**已知结局（多周目测试），确认最后一个结局解锁后，卡片状态正确切换为"已阅尽此世"
4. 章节阅读页脉络线渲染正确：已走路径为实线，当前点高亮，前路自然淡出不画具体分岔，若该局经历过选择则有一条淡出的支流曲线
5. 章节切换过渡动效体验流畅，无生硬跳转感
6. 检查所有玩家可见文案，确认没有泄露 `chapter`/`flag`/`exits` 等内部工程术语

## 八、前端框架复杂度评估（收尾）

完成上述UI改版后，基于实际交互复杂度评估以下问题，产出简短的评估结论（不要求本阶段执行任何框架迁移）：

- 脉络线组件、章节过渡动效、卡片进度状态等交互，用当前 Alpine.js（v1.3已引入，~15KB量级）是否已经出现明显的状态管理吃力或代码可维护性下降的迹象？
- 如果后续（v2.1+）要支持"AI自生长分支"或更复杂的多存档管理，是否预判会显著超出 Alpine.js 的合理适用范围？
- 结论建议以下三选一：①维持现状，Alpine.js足够；②建议在下一个内容形态大改动前评估迁移到Vue3（说明触发迁移的具体信号是什么，而非现在预判）；③需要更多实际使用数据才能判断，建议延后到v2.1启动前再评估一次

该评估结论写入文档末尾即可，作为v2.1规划的输入参考，不在本阶段做任何强制决策。

### 评估结论（v2.0.3 落地后）

**结论：①维持现状，Alpine.js足够。**

**本阶段实际复杂度观察：**
- 选择页模式切换 + 长镜卡片仍是 vanilla `app.js` 的 DOM 拼装；状态只有 `playMode` / `pendingSaveId` 等少量字段，没有出现跨组件双向同步压力。
- 阅读页新增的脉络线是纯函数 `buildSpineSvg(visitedCount, hadBranch)` → `x-html`，章节过渡是两个布尔/标题字段 + `setTimeout`；Alpine 组件内状态面仍清晰，未出现深层嵌套 store 或大量 `watch` 需求。
- 长镜与特写共用同一 Echoes 壳子，差异靠 `workType` 开关，复杂度增量主要在后端 enrichment，不在前端状态图。

**对 v2.1+ 的预判（不触发现在迁移）：**
- AI自生长分支若仍是「接口返回摘要状态、前端只渲染」的模式，Alpine 仍够用。
- 若出现以下信号，再在内容形态大改动前评估 Vue3（或同类组件框架）：①多存档槽位需要可编辑的会话列表 + 实时同步；②脉络线变成可交互探索（悬停/回看/多时间线对照）且状态与对话流强耦合；③同一页多实例阅读壳（编辑预览、对照游玩）导致 Alpine 组件边界反复泄漏。

**说明：** 当前可维护性的主要风险不在 Alpine，而在选择页仍用命令式 DOM 与阅读页 Alpine 双轨并存；短期可接受，若 v2.1 大改 UI，再考虑是否统一到单框架，而非仅为长镜 MVP 提前迁移。
