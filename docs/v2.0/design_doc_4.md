# 入戏 v2.0.detail-4 —— 长镜内部交互设计 + 账号体系

> 本文档在 v2.0 总纲三份detail文档基础上新增，处理三个此前未覆盖的需求：①章节切换需要停顿在"结算画面"，玩家手动确认后才进入下一章；②引入账号体系，覆盖**特写与长镜两种模式**，支持存档持久归属、跨设备找回、个人游玩历史；③账号体系为**可选**而非强制——两种模式均保留匿名游客可玩的路径，登录只是获得"记录与找回"能力的方式。本文档同时包含**对 detail-2、detail-3 的修订说明**，两份文档中与本次新增需求冲突的部分以本文档为准。

---

## 一、本阶段目标

1. 章节切换流程从"自动判定并直接生效"改为"判定完停顿展示结算画面，玩家手动确认后才真正推进"
2. 完整设计长镜模式内部的交互状态机与各画面构成（对话中 / 章节结算 / 作品终局结算）
3. 引入账号体系，**覆盖特写与长镜两种模式**：登录用户可获得存档持久化（长镜）、跨设备找回、多周目"已发现结局"记录、以及**个人游玩历史**（两种模式共通）
4. **两种模式均保留匿名游客路径**：不登录也能完整体验——特写维持现状不变；长镜支持"单次体验"（游客可以完整玩完一部长篇作品，但不保存进度、关闭后无法找回，也不计入历史）
5. 登录与否**不影响能否游玩**，只影响"游玩痕迹是否被记住"，这是demo阶段的核心产品取舍：优先保证低门槛试玩，账号是增值能力而非准入门槛

## 二、本阶段明确不做

- 不做强制登录门槛——两种模式在任何情况下都不应该因为"未登录"而无法开始游玩
- 不做社交类功能（好友、排行榜、战绩分享）
- 不做多存档槽位（登录用户仍是单存档位，只是归属从"设备/session"升级为"账号"）
- 不做复杂账号找回流程（短信验证码等），走邮箱最小化实现
- 不做游客身份到账号的"事后绑定/合并"（比如游客先玩了一半，中途登录，把游客期间的临时进度带入账号）——本阶段游客与账号是两条独立路径，登录即视为从账号已有存档（或全新）开始，不做合并逻辑，避免增加复杂度

## 三、章节切换交互流程（核心新增）

### 3.1 状态机

```
对话进行中
  → 本轮导演输出 game_over=true（含 chapter_wrap_up）
  → 【章节结算画面】（新状态，停顿，等待玩家主动确认）
  → 玩家点击"翻到下一章"
  → 引擎正式判定并写入 current_chapter_id
  → 若判定出下一章：新章节过渡态 → 回到"对话进行中"
  → 若判定为终点章节：进入【作品终局结算画面】
```

**关键变化**：detail-2原设计中，"判定出口"和"存档更新为下一章"是同一步完成、直接对前端生效的。本次需要拆成两步——**判定**（引擎算出应该去哪一章，但先不写入存档的 `current_chapter_id`，只作为待确认的预备值返回给前端）与**确认**（玩家在结算画面点击按钮后，才真正调用接口把存档推进到下一章）。这是对 detail-2 第四节的修订，见第七节详述。

### 3.2 章节结算画面

触发时机：本章 `game_over=true` 判定发生后，不直接跳转下一章，先展示此画面。

构成元素（自上而下）：

| 元素 | 说明 |
|---|---|
| 完成态标记 | 章节标题 + "终" 或类似的完结字样 |
| 剧情脉络线（更新版） | 追加本章节点到脉络线上；若本章存在多条exits而本局只走了一条，在此绘制那条淡出的支流（延续detail-3已定的视觉语言） |
| 数值变化摘要 | **只展示本章实际变动的维度**（即本章key_points/pitfalls涉及到的维度集合，未变动的维度不重复罗列），呈现方式如"信任 62 → 70"，聚焦"这一章发生了什么变化"而非重复展示全部5维度 |
| 章节摘要文案 | 即 `chapter_wrap_up.summary`，以正文级别的文学化文字呈现，不是调试信息 |
| 主操作按钮 | "翻到下一章"，强调色实底按钮，无次要操作——长镜是单向推进的阅读体验，结算画面不需要"返回"选项 |

**不展示的内容**（刻意排除，避免暴露机制）：本章触发了哪些flag、待命中的key_points/pitfalls细节、是否存在其他未走的exits候选——这些都属于内部判定依据，摘要文案已经承载了"发生了什么"的文学化信息，不需要额外的机制说明。

### 3.3 作品终局结算画面

触发时机：判定出的下一章是终点章节（无exits）。

构成元素，复用v1.0已有的结局卡片设计语言，适配长镜调性：

| 元素 | 说明 |
|---|---|
| 结局氛围色 | 由该终点章节的 `ending_tone` 字段决定（新增schema字段，见第六节），如正面用强调色 `#4B6355`、遗憾/悲剧用 `#8A4D4A` |
| 结局正文 | 终点章节自身的内容即为结局正文（复用章节的background/opening_line承载，不需要额外的ending_text字段——长镜的终点章节本身就是完整的一章内容，不同于短局的简短结局文案） |
| 全局数值回顾 | 贯穿整部作品的最终5维度数值 |
| 完整脉络线 | 本周目走过的完整路径，作为"足迹"总览，带纪念感 |
| 结局解锁提示 | 若该账号首次到达此终点章节——展示"你发现了一个新的可能"这类文案；若此前已到达过，不做特别提示（不用每次强调"已知结局"） |
| 操作按钮（两个） | "再入此局"（同作品重开新周目，从entry_chapter开始新存档）、"返回入口"（回选择页） |

## 四、账号体系设计

### 4.1 范围与核心原则

**账号体系覆盖特写与长镜两种模式，但两种模式在未登录状态下都必须保持完整可玩。** 登录只解锁"记录与找回"能力，不作为游玩的前置条件：

| | 游客（未登录） | 登录用户 |
|---|---|---|
| 特写 | 完整可玩，行为与现状完全一致 | 完整可玩，额外记录到个人游玩历史 |
| 长镜 | 完整可玩，支持"单次体验"——可以从头玩到某个结局，但不落地为可找回的持久存档；关闭页面或较长时间未操作后进度不保证能恢复 | 完整可玩，存档持久化、跨设备可续玩、结局解锁记录持续累积 |

### 4.2 认证方式

- 使用 **Supabase Auth**，与已经引入的 Supabase Postgres 同源集成，不引入额外的第三方服务
- 推荐 **邮箱 magic link**（免密码，轻量，Supabase Auth原生支持）作为主要登录方式
- 是否额外接入一个OAuth快捷登录（如Google）视实现时间成本决定，非本阶段必须项

### 4.3 触发时机（不设强制门槛，仅做柔性引导）

- **全局入口**：页面固定位置（如顶部角落）常驻一个轻量的"登录"入口，任何时候都可以主动点击登录，不依赖特定触发场景
- **长镜进入时的柔性提示**：玩家以游客身份点击进入某个长镜作品时，展示一条**可关闭、不阻塞**的提示，如"游客模式下进度不会被保存，登录后可随时继续"，玩家可以选择登录，也可以直接关闭提示继续以游客身份游玩
- **长镜结算画面的柔性提示**：游客身份完成某个结局后，结算画面可以附带一条不强调的引导（如"登录后这段旅程会被记住"），同样可关闭、不阻塞返回选择页或重玩的操作
- **特写不做任何登录提示的强插入**：特写体验维持现状，若要在结局卡片上顺带加一条不打扰的"登录后可查看游玩记录"提示，需保证不改变现有结局卡片的核心视觉与操作流程
- 登录成功后，若当前正处于某个长镜作品的游玩过程中，直接在原有进度基础上转为持久存档（不重置、不要求重新开始）

### 4.4 数据模型变化

- `saves` 表新增 `user_id` 字段，**允许为空**：游客游玩时 `user_id` 为空，存档仅在当前浏览器session范围内维持"单次体验"的连续性（比如中途刷新页面不丢，但不保证长期持久或跨设备），登录用户的 `user_id` 关联 Supabase Auth 用户，享受完整持久化
- 存档归属：登录用户为"同一账号 + 同一work只维护一份存档"（v2.0单存档位约束延续）；游客存档不做跨会话归属，浏览器session结束后视为丢失，这是"单次体验"的预期行为，不是需要修复的问题
- `discovered_endings` 仅对登录用户按"账号+work"维度持久化；游客的结局发现状态不跨会话保留，每次以游客身份重新进入都是"仍有未至之地"的初始状态
- **新增 `short_play_records` 表**（特写游玩历史，仅登录用户产生记录）：记录 `user_id`、`work_id`、`outcome`（胜利/失败）、完成时间。游客完成特写不产生此记录

### 4.5 个人游玩历史（新增能力）

面向登录用户的轻量能力，不需要做成完整的"账号中心"页面，一个简单的列表视图即可：

- **特写记录**：已完成的作品列表，展示标题、达成结局（胜利/失败）、完成时间
- **长镜记录**：进行中/已完结的作品列表，展示标题、当前进度（如"第3/6章"）、已发现结局数量（这里可以展示具体数字，因为这是玩家自己的记录，不是对外的"神秘感"展示位——神秘感只针对未探索前的公开卡片信息，玩家查看自己的历史时应该看到明确的个人数据）
- 入口：登录后从第4.3节的常驻登录入口位置切换为"我的记录"入口

## 五、后端接口变化

- 认证本身大部分由 Supabase Auth SDK 直接处理
- **鉴权改为可选而非强制**：长镜与特写相关接口均支持"带JWT"和"不带JWT"两种请求。带JWT时取出 `user_id` 用于存档归属判定与历史记录写入；不带JWT时按游客路径处理（长镜走session级临时存档、特写不产生历史记录），**接口不应因未鉴权而拒绝请求**
- **新增独立的"确认进入下一章"接口**，例如 `POST /api/works/{work_id}/chapters/advance`：前端在章节结算画面点击"翻到下一章"后调用，此时才真正把存档的 `current_chapter_id` 更新为判定出的下一章、清空章节内状态（对应detail-2第四节步骤5），并返回新章节数据。原本判定出口的那次响应（对应detail-2第四节步骤4）**只返回结算画面所需数据和预备的下一章id，不直接生效**。游客场景下同样遵循这套两步流程，只是数据落地方式不同（session级 vs 账号级持久化）
- **新增 `GET /api/me/history` 接口**（需鉴权）：返回登录用户的特写完成记录与长镜进行中/已完结记录，供第4.5节"个人游玩历史"使用

---

## 六、Schema 新增字段

- 终点章节（无exits的章节）新增 `ending_tone` 字段：枚举值 `positive` / `bittersweet` / `negative`（可按实际内容需要扩展），用于结算画面配色
- `saves.user_id`：允许为空（nullable），空值代表游客场景
- 新增 `short_play_records` 表：`user_id`（非空，仅登录用户产生记录）、`work_id`、`outcome`、`completed_at`

## 七、对已有detail文档的修订说明

### 对 detail-2 的修订

1. **第四节步骤4-6**：原设计中判定出口后直接更新存档、一次性返回结果。修订为拆分两步——判定接口（本轮对话消息接口，触发`game_over=true`时）只返回"本章结算所需数据+预备的next_chapter_id"，**不写入**存档的`current_chapter_id`；真正的存档推进只发生在玩家调用第五节新增的 `advance` 接口时。短局场景不受此修订影响（短局的game_over直接对应最终结局，没有"下一章"的概念，原有行为保持不变）。
2. **第六节存档读写接口**：原文"是否每局都落库、还是仅在需要断线续玩时才落库，由实现时视性能取舍决定"，修订为**明确要求**——长镜场景下，每一轮对话结束后都必须落库（登录用户落到账号存档，游客落到session级临时存档），不作为可选的性能取舍项。短局场景的存档落库策略不受此修订影响，仍可按原实现取舍处理，但若玩家已登录，完成结局时需额外写入一条 `short_play_records` 记录（不改变短局本身的游玩机制与响应结构，只是后端多一步"若已登录则记一笔历史"的旁路逻辑，游客身份下跳过此步骤）。

### 对 detail-3 的补充

- 需要补充**账号登录引导UI**：简洁的邮箱输入+确认界面，走Supabase Auth自带流程，视觉上延续"世界残响"整体调性（衬线字体、米色底、强调色按钮），不需要额外设计复杂表单或独立的账号中心页面
- 需要补充**常驻登录入口**（页面固定位置，如顶部角落）与**个人游玩历史列表**的最小化视觉方案，两者都应保持轻量，不需要按"完整账号中心"的复杂度设计
- 第五节"章节切换过渡"原描述的"渐隐+新章节标题浮现"这一简短过渡，修订为本文档第三节的完整"章节结算画面"停顿流程，不再是单纯的动效过渡

## 八、验证要求

1. 完整走通一次长镜demo作品：每章结束后正确停在结算画面，展示本章数值变化摘要、脉络线更新、摘要文案；点击确认后才真正进入下一章，中途不点击确认不会自动推进
2. 验证"确认进入下一章"接口调用前，存档的 `current_chapter_id` 确实未变；调用后才正确更新
3. 验证游客路径：未登录状态下能完整体验长镜（从进入到某个结局），过程中不阻塞、不强制弹窗登录，柔性提示可正常关闭；确认游客的进度不产生跨会话持久化（预期行为）
4. 验证账号登录流程：登录引导正常触发（长镜进入/结算画面的柔性提示、常驻登录入口均可正确唤起），登录成功后正确创建/恢复账号级存档；若登录发生在游玩过程中，能在原有进度基础上转为持久存档
5. 验证跨设备场景：同一账号在设备A玩到第2章后登出，在设备B重新登录，能正确恢复到第2章的进度
6. 验证多周目场景：完成一个结局后选择"再入此局"，新周目独立计算数值/flag，但该账号的 `discovered_endings` 记录保留之前发现的结局
7. 验证作品终局结算画面的"首次发现"提示逻辑：首次到达某终点章节时展示提示，非首次到达时不展示
8. 确认特写模式**无论登录与否**，核心游玩流程与现状完全一致，不因账号体系引入而改变体验；额外验证登录用户完成特写后，`short_play_records` 正确写入一条记录，游客完成特写不产生记录
9. 验证"个人游玩历史"：登录用户能看到自己已完成的特写记录（标题/结局/时间）与长镜进度记录（含具体已发现结局数量，此处不做神秘感隐藏）
10. 每轮对话结束后检查数据库，确认存档（账号级或游客session级）确实被实时写入，而非仅在章节切换时才落库

## 九、给后续阶段的说明

本文档完成后，长镜模式的核心交互体验（含章节结算停顿）与覆盖两种模式的账号体系（含游客路径、个人游玩历史）完整闭环。若后续需要多存档槽位、游客进度到账号的绑定合并、更完整的账号中心（跨作品统计、成就系统等）能力，属于独立事项，不在v2.0范围内预先设计。
