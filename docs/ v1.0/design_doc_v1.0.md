# 入戏 v1.0 —— Multi-Agent 架构重构指导

## 本版本目标

将当前"单一LLM生成对话+数值"的模式，拆分为**导演agent + 演员(roleplay)agent** 的双模型协作结构，并建立**显式的prompt模板管理机制**，为后续所有玩法迭代打好地基。

**本版本不做**：难度分级、提示/求助机制、开发者调试面板、前端框架升级——这些安排在 v1.1/v1.2/v1.3，见文末路线图。

---

## 一、为什么要拆分成两个 Agent

当前单模型同时承担"扮演角色说话"和"判断数值变化"两个职责，容易出现：
- 角色对话为了"演出感"而口语化、模糊，导致数值判定的输入依据不稳定
- 数值变化和角色台词的情绪不一致（比如愤怒值涨了但台词读起来很平静）
- 难以分别优化"演技"和"判定准确性"这两件事的prompt

拆分后职责清晰：

| Agent | 输入 | 输出 | 职责 |
|---|---|---|---|
| **导演 Agent** | 剧本设定、当前数值、历史对话摘要、玩家最新发言、胜负条件 | 结构化JSON：数值变化、对本轮角色反应的**指导性描述**、是否结束及结局判定 | 裁判 + 剧情调度，不产出角色台词 |
| **演员 Agent** | 角色人设、历史对话、玩家发言、**导演给的本轮指导** | 结构化JSON：角色的对话回复文本 | 只负责"演"，接受导演调度，不做数值判断 |

调用顺序：**先导演，后演员**。导演先根据玩家发言判定数值变化和"这轮角色该往什么方向反应"，再把这个方向性指导传给演员，演员据此生成具体台词——保证数值和台词的情绪始终一致。

## 二、结构化输出规范（JSON Schema）

### 导演 Agent 输出格式

```json
{
  "stat_changes": {
    "怀疑值": -5,
    "愤怒值": 10
  },
  "reaction_instruction": "唐晶感到罗子君在回避问题，语气应转为更加不耐烦和追问式，但还未到爆发点",
  "game_over": false,
  "outcome": null,
  "ending_text": null
}
```

- `stat_changes`：本轮各数值的**增量**（不是绝对值），key 必须和剧本定义的数值名一致
- `reaction_instruction`：**自然语言指导**，描述角色本轮该表现出的情绪/态度倾向，供演员agent参考，不是台词本身
- `game_over`：布尔值，本轮是否触发结局
- `outcome`：`"win"` / `"lose"` / `null`（未结束时为null）
- `ending_text`：结局文案，仅在 `game_over=true` 时填写，否则为 `null`

### 演员 Agent 输出格式

```json
{
  "reply": "角色本轮说的话，纯对话内容，不包含任何旁白或元信息"
}
```

保持简单——演员agent只做一件事，输出格式故意做到最小化，减少解析出错概率。

### 解析容错要求

两个agent都要求LLM"只返回JSON，不要有任何前后缀说明或markdown代码块标记"，但仍需后端做防御性解析：
- 解析失败时的兜底策略：导演解析失败 → `stat_changes`全部记为0、`game_over=false`；演员解析失败 → 用一句预设的"角色一时语塞"类兜底文案
- 记录解析失败日志（打印到控制台即可，demo阶段不需要专门的日志系统），方便你后续观察哪类输入容易导致格式错误，用于反哺prompt调整

## 三、Prompt 模板显式管理

新建 `prompts/` 目录，每个agent的prompt作为独立的模板文件管理，不要把prompt硬编码在业务逻辑代码里。

```
prompts/
├── director_system.txt      # 导演agent的system prompt模板
├── director_user.txt        # 导演agent每轮拼接的user prompt模板
├── roleplay_system.txt      # 演员agent的system prompt模板（含角色人设占位符）
└── roleplay_user.txt        # 演员agent每轮拼接的user prompt模板
```

模板中用 `{{变量名}}` 标记占位符（简单字符串替换即可，demo阶段不需要引入Jinja2），例如 `director_user.txt`：

```
【剧本背景】
{{background}}

【当前数值状态】
{{current_stats}}

【历史对话】
{{conversation_history}}

【玩家本轮发言】
玩家（{{player_character_name}}）：{{player_message}}

【胜负条件】
胜利条件：{{win_condition}}
失败条件：{{lose_condition}}
当前轮次：{{current_turn}} / {{max_turns}}

请根据以上信息，判断本轮数值变化、角色应有的反应方向，以及是否触发游戏结束。只返回JSON，不要包含任何其他说明文字。
```

在 `game/` 目录下新增 `prompt_manager.py`，提供统一的加载和填充接口：

```python
# game/prompt_manager.py 的职责（不要求具体实现，描述接口行为）
# - load_template(template_name) -> 读取 prompts/ 下对应文件内容
# - render(template_name, **kwargs) -> 读取模板并替换占位符，返回最终prompt文本
# - 所有 llm_client.py 中调用LLM前的prompt组装，都必须通过这个模块完成，
#   禁止在 engine.py 或 llm_client.py 里直接拼接prompt字符串
```

**这样做的目的**：你以后调优某个agent的表现，只需要打开对应的 `.txt` 文件改文字，不需要碰任何 Python 代码逻辑，减少调优时误改逻辑代码的风险。

## 五、接口层的变化

`POST /api/session/message` 的**外部接口不需要变**（前端调用方式不变），但内部处理流程调整为：

1. 读取当前 session 状态（历史、数值、轮次）
2. 调用 `director.judge(...)`，传入剧本设定+当前状态+玩家发言，得到 `stat_changes`、`reaction_instruction`、结束判定
3. 更新 session 中的数值状态（按 `stat_changes` 累加，并 clamp 在剧本定义的 min/max 范围内）
4. 若 `game_over=true`：跳过演员调用，直接返回 `ending_text` 作为本轮展示内容
5. 若未结束：调用 `roleplay.respond(...)`，传入角色人设+历史对话+导演的 `reaction_instruction`，得到角色台词
6. 将角色台词存入历史，返回给前端：`{reply, stats, turn, game_over, outcome, ending_text}`

前端展示上，`reaction_instruction` **不展示给玩家**（这是幕后导演指令，暴露出来会破坏沉浸感），只用于内部调用链。

## 七、UI视觉规范（配合示意图使用）

以下规范对应另附的两张UI示意图（对局界面、结局结算卡片），用于本版本前端实现时的具体参照。**颜色数值以本节为准**，示意图用于理解布局结构。

### 整体容器
- 对话卡片：圆角 16px，背景为浅色卡片背景（不用纯白，用极浅灰米色调），边框用 0.5px 极细描边，不用阴影
- 顶栏（剧本标题+目标+轮次）：固定在对话区顶部，用 0.5px 底边框与对话区分隔，不需要额外背景色区分

### 数值条
- 每个数值：标签 + 数字（12px，灰色次要文字色）在上方，进度条在下方
- 进度条：高度 6px，圆角 3px，底色为浅灰，填充色**必须按数值语义区分**：
  - 负面/危险类数值（愤怒值、怀疑值达到高位等）：高数值时用红色系（`#E24B4A`），中等用黄褐色系（`#BA7517`）
  - 正面/友好类数值：用绿色系
  - 具体映射由你在剧本JSON里为每个数值定义"方向性"（越高越好 or 越低越好），前端根据方向性和当前数值区间动态选择颜色档位（比如 <30% 用绿、30-70% 用黄、>70% 用红，如果是"越高越好"的数值则反过来）
- 数值变化时：宽度变化要有 `transition: width 0.4s ease` 的平滑过渡，不需要额外的闪烁提示（demo阶段简化，动画本身已经足够传达变化）

### 对话气泡
- 左侧（AI角色）：浅灰背景卡片色，圆角 14px 但贴近发言方的角落为 4px（即左侧气泡左下角4px，右侧气泡右下角4px，模拟聊天软件的"尖角朝向"）
- 右侧（玩家）：用强调色系背景（如紫色 `#7F77DD` 配对应的浅色文字），与AI气泡形成明显视觉区分
- 角色名：11px，灰色次要文字色，显示在气泡上方（AI侧）或下方（玩家侧，靠右对齐）
- AI"思考中"状态：用三个跳动的小圆点占位，不需要文字提示

### 提示/求助按钮（"请帮帮我"）
- 位置：输入框上方，独立一行
- 样式：普通次要按钮样式（不用强调色，不抢视觉焦点），配一个提示类图标（如 `ti-help-circle`）
- 这是本版本UI需要预留的交互位，具体触发逻辑在 v1.1 实现，v1.0 阶段可以先做成按钮存在但点击后暂时无实际效果（或提示"即将上线"），重点是把布局位置定下来

### 结局结算卡片
- 以居中弹层形式覆盖对话区（半透明黑色遮罩 + 居中白色卡片），不用浏览器原生 `alert`
- 卡片内容自上而下：结果图标（胜利用打勾图标+绿色圆底，失败用叉图标+红色圆底）→ 轮次+结果一句话小字 → 结局标题（如"唐晶原谅了你"）→ 结局文案（2-3句） → 最终数值回顾（每个数值一个小卡片，展示最终数值） → 操作按钮（"再玩一次"、"换一个剧本"，两个按钮都用普通样式，不强调优先级）

### 颜色使用原则
- 全局只使用 2-3 个语义色（红/黄/绿 用于数值状态，紫色系或你选定的一个强调色用于玩家消息气泡），不要引入更多颜色造成视觉噪音
- 文字在彩色背景上时，使用该颜色色系里更深的色阶而非纯黑色（比如浅绿背景配深绿文字，不要浅绿配黑字）
- 圆角统一：卡片用 12-16px，按钮和小组件用 8px 左右，气泡用 14px（贴角处 4px）

## 八、验证要求

完成后需验证：
1. 用现有的 `example_script.json` 跑通完整对局，确认导演和演员的调用顺序、数值更新、结局触发都符合预期
2. 人为构造几个"容易让LLM返回非JSON内容"的边界输入（比如让玩家发一段很长的话，或者发无意义乱码），确认解析失败时兜底逻辑生效，游戏不会崩溃
3. 打印/检查两个agent各自收到的最终prompt文本（可以临时加日志），确认 `prompt_manager.py` 的模板替换是正确的，没有变量遗漏或格式错乱
4. 部署到 Render 验证公网可用，行为和本地一致
5. 对照"UI视觉规范"章节逐项检查前端实现：数值条颜色是否随数值区间正确切换、宽度变化是否有平滑过渡、对话气泡左右区分是否清晰、结局卡片是否为居中弹层而非原生alert

## 后续路线图（本次不实现，仅供了解走向）

- **v1.1 玩法节奏细化**：剧情提示机制（如"请帮帮我"按钮，触发一次额外的LLM调用给玩家隐晦提示而不直接剧透）、难度分级（不同剧本可配置不同初始数值/轮数上限/结束条件宽松度，作为剧本JSON里的"难度变体"字段）
- **v1.2 开发者调试模式**：单独的 `/dev` 路由或页面，可视化编辑当前剧本的prompt模板、初始数值、胜负条件，并支持"沙盒模式"直接试玩验证改动效果，无需重启服务
- **v1.3 前端框架升级与UI精细化**：评估引入轻量前端框架（如 Alpine.js 或 Vue 3 CDN版本，仍然免构建）替代当前原生JS，以更好地支持组件化的对话气泡、数值条动画、多剧本卡片式选择页等更复杂的交互
