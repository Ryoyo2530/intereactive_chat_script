# 入戏 v1.1（第一优先模块）—— 剧本分类标签系统 + 选择页展示

## 在v1.1路线图中的位置

v1.1原计划内容为：剧情提示机制（"请帮帮我"）+ 难度分级。**本次新增的剧本分类与展示系统，作为v1.1的第一优先模块**，先于提示机制和难度分级实现，原因：
- 分类系统影响剧本JSON schema和 `/api/scripts` 接口结构，后续难度分级也会往剧本schema里加字段，两者最好先后有序，避免schema反复变动
- 分类展示页是"多剧本平台"的入口体验，随着剧本数量增加（影视改编/原创剧情等类型陆续上线），越早有分类导航，测试和分享给朋友时的体验越完整

---

## 一、分类标签设计：两个独立维度

不做单一层级分类（如"影视区/原创区"二选一的目录结构），而是用**两个正交标签维度**，因为"剧本来源"和"剧本主题"是两件独立的事：

| 维度 | 说明 | 取值方式 | 当前枚举值（可扩展） |
|---|---|---|---|
| `origin_tag` | 剧本内容的来源 | 单选 | `影视同人` / `原创剧情` |
| `theme_tags` | 剧本讨论的关系/冲突领域 | 可多选（但建议每个剧本主推1个作为首要展示） | `恋爱` / `职场` / `家庭` / `友情` / `社交` |

两个维度的枚举值本身也应该是可扩展的（不要硬编码在前端逻辑里），新增剧本如果用到新的主题标签，直接在剧本JSON里加，前端标签渲染逻辑应能自动兼容未预定义颜色的新标签（见下方"未定义标签的兜底样式"）。

---

## 二、剧本 JSON Schema 扩展

在 `scripts/*.json` 中新增以下字段：

```json
{
  "id": "example_script",
  "title": "我的前半生·唐晶篇",
  "origin_tag": "影视同人",
  "theme_tags": ["友情"],
  "teaser": "罗子君与贺涵的关系让唐晶心生芥蒂，你该如何解释才能挽回这段友情？",
  "player_role_hint": "扮演 罗子君 · 对手 唐晶",
  "estimated_turns_hint": "约 10-15 轮",
  "...": "（其余字段如background/ai_character/stats/key_points等保持v1.0结构不变）"
}
```

字段说明：
- `teaser`：选择页展示的一句话简介，**独立于**导演/演员prompt里用的 `background`（background给LLM看的是完整剧本设定，teaser是给玩家看的营销向短句），两者不要混用同一字段
- `player_role_hint`：选择页展示"扮演谁 · 对手是谁"，从 `player_character.name` 和 `ai_character.name` 拼接即可自动生成，也可以在剧本里单独配置以支持更灵活的措辞（比如"扮演 你自己"这种不直接用角色名的情况）
- `estimated_turns_hint`：展示用的文案，和 `max_turns` 数值分开维护（避免"约10-15轮"这种模糊表述和精确的轮次上限强绑定，未来调整 `max_turns` 不需要同步改文案，反之亦然）

## 三、接口层变化

`GET /api/scripts` 返回结构扩展，需要包含分类和展示所需的全部字段：

```json
[
  {
    "id": "example_script",
    "title": "我的前半生·唐晶篇",
    "origin_tag": "影视改编",
    "theme_tags": ["恋爱"],
    "teaser": "...",
    "player_role_hint": "扮演 罗子君 · 对手 唐晶",
    "estimated_turns_hint": "约 10-15 轮"
  }
]
```

注意：**不要**把 `background`、`key_points`、`stats` 等仅供后端LLM调用使用的字段暴露在这个接口里——这些内容如果直接展示给玩家或者被玩家在浏览器里看到源数据，会剧透游戏机制或判定逻辑。`/api/scripts` 只返回选择页展示所需的字段。

## 四、前端展示要求（配合另附的选择页示意图）

### 顶部筛选栏
- 两行标签筛选按钮：第一行是 `origin_tag` 筛选（全部/影视改编/原创剧情，单选，默认"全部"高亮），第二行是 `theme_tags` 筛选（可多选，不选默认展示全部）
- 筛选按钮样式：胶囊形状（圆角999px），未选中为透明背景+细边框，选中为实底背景，不需要额外的动画效果

### 剧本卡片
- 每张卡片：顶部一行标签徽章（`origin_tag` + 首要 `theme_tag`，各自用不同底色区分，同一类维度用统一色系，比如所有"影视改编"徽章都用同一个色，不同剧本不应各自随意配色）
- 标题（15px，中等字重）
- 一句话简介（`teaser`，12.5px，次要文字色，1-2行）
- 底部一行：左侧显示 `player_role_hint`，右侧显示 `estimated_turns_hint`，都用11px小字灰色
- 卡片本身：圆角14px，浅色背景卡片，细边框，不需要阴影或悬浮效果（demo阶段简化）

### 未定义标签的兜底样式
- 前端应维护一个"标签→颜色"的映射表（可以是简单的JS对象），包含当前已知的枚举值配色
- 遇到映射表中不存在的新标签值（比如你新加了一个 `theme_tag: "悬疑"` 但前端还没更新映射表）时，使用统一的中性灰色兜底样式，而不是报错或不显示——保证新剧本上线时即使前端样式没来得及跟上，功能也不会坏

---

## 五、验证要求

1. 在 `scripts/` 目录下新增至少两个占位剧本（可以复用/简化现有的裁判逻辑，内容随意但要覆盖不同的 `origin_tag`/`theme_tags` 组合），确认选择页能正确展示三种及以上剧本卡片
2. 验证筛选栏交互：切换 `origin_tag` 筛选和勾选/取消 `theme_tags` 筛选，卡片列表能正确联动过滤
3. 验证 `/api/scripts` 返回内容不包含 `background`、`key_points`、`stats` 等内部字段（可以直接在浏览器开发者工具的网络面板里检查接口返回的原始JSON）
4. 人为构造一个使用未在前端映射表中定义的新 `theme_tag` 的剧本，确认前端能用兜底灰色样式正常展示，不报错
5. 部署到 Render 验证公网表现一致

## 后续可能的延伸（本次不实现）

- 分类维度可能后续再加一个"情绪基调"标签（如轻松/虐心/搞笑/严肃），用于进一步细化选择页的浏览体验，是否需要视剧本数量增长情况再决定
- 难度分级完成后，卡片上可以再加一个难度展示位（如星级或文字标签），与本次的标签系统并列展示，届时只需扩展卡片布局，不需要改动本次的标签架构
