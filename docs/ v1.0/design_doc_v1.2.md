# 入戏 v1.2 —— Dev模式 + 剧本验证沙盒

## 本版本目标与在v1整体里程碑中的位置

让你能够**在线编辑剧本内容**（标签、人设、关键点、数值、prompt相关字段等）并在发布前**验证效果**，不需要每次调整内容都去改代码、走一次git push + 重新部署的流程。

本版本做完后，v1阶段的整体目标基本达成：
- 稳定的多agent工程框架（导演/演员分离、结构化输出、关键点驱动数值）
- 还算美观的玩家端UI（对局界面、数值条、情绪标签、结局卡片、剧本分类选择页）
- **内容层面可以独立于代码迭代**（本版本要实现的核心能力）

更精细的打磨（表单化编辑器、更复杂动效、难度分级的深度呈现）留给v2.0，本版本刻意不做。

**本版本不做**：剧本表单化编辑（先用原始JSON文本编辑器，表单化留给v2.0）、多用户权限管理、数据库迁移、难度分级功能本身（如果尚未实现，仍按原计划单独排期，不在本次范围内）。

---

## 一、整体设计思路：内容与工程分离

核心思路是让 `scripts/*.json` 从"部署时固化的静态文件"变成"运行时可读写的内容层"：

- 剧本由后端**运行时加载**，dev模式保存后**立即对所有访客生效**，不需要重启服务
- **持久化的现实边界**：Render 免费层的文件系统在服务重新部署/重启后会重置。因此dev模式的编辑在当次运行期间是"活的"，但要让改动**跨部署持久保留**，你需要定期点击"导出剧本"把内容下载下来，手动提交回GitHub仓库——这是一个刻意保留的手动步骤，用来避免为demo规模引入数据库或持久化磁盘这类额外基础设施
- 如果未来内容迭代频率高到手动导出很麻烦，可以在v2.0考虑升级到数据库或Render的持久化磁盘，本版本不做

## 二、访问控制

- 新增环境变量 `DEV_MODE_PASSWORD`
- `/dev` 路由展示一个简单密码输入表单，校验通过后设置一个session token（cookie即可）
- 所有 `/api/dev/*` 接口都必须校验该token，未授权请求一律拒绝
- 不需要做多用户账号体系，这是给你自己用的单人访问口令

## 三、Dev模式页面结构与功能

### 剧本列表管理
- 展示所有剧本，每项可操作：编辑 / 复制为新剧本 / 删除（删除需二次确认弹窗）
- "新建剧本"入口：提供一份最简JSON模板（复用 `example_script` 结构）作为起点，避免从零手写

### 剧本编辑器（本版本先做原始JSON编辑，表单化留v2.0）
- 一个大文本框承载完整剧本JSON，可直接编辑
- 编辑器旁提供字段说明的静态参考文本（列出当前schema支持哪些字段及其作用），避免你每次都要翻文档去记字段名
- "验证"按钮：触发静态schema校验（见第四节），不保存
- "保存并发布"按钮：仅在验证通过后可点击，点击后写入实际的 `scripts/{id}.json`，**立即对玩家生效**
- "导出全部剧本"按钮：打包当前所有 `scripts/*.json`，供你下载后手动提交回git仓库

### 试玩验证面板（本版本的核心"验证模块"）
- 选择任意一个剧本（草稿或已发布均可），在dev模式内开启一局仅自己可见的测试对局
- 每轮对话展示两部分：
  1. 玩家视角看到的最终效果（角色台词、情绪标签、数值条变化）
  2. **可折叠展开的调试信息**：导演Agent的完整原始JSON输出（`hit_key_points`/`hit_pitfalls`/`stat_changes`/`reaction`）、演员Agent的完整原始JSON输出（`reply`/`emotion_tag`）
- 这样你能直接确认"这句话有没有命中预设的关键点""数值增量是否符合预期""情绪标签判断是否准确"，不需要靠观察最终文字去反推内部逻辑

## 四、静态Schema校验规则（"验证"按钮触发）

校验项：
- 必填字段是否齐全：`title`、`origin_tag`、`theme_tags`、`background`、`ai_character`、`player_character`、`stats`、`key_points`、`win_condition`、`lose_condition`、`max_turns`、`opening_line`
- `stats` 中每个数值的 `initial` 是否落在 `min`~`max` 区间内
- `key_points`/`pitfalls` 的 `id` 在同一剧本内是否唯一
- `key_points`/`pitfalls` 的 `hit_stat_changes` 里引用的数值名，是否都在 `stats` 中有定义
- `win_condition`/`lose_condition` 字符串是否能被现有的条件解析逻辑正确解析，不报语法错误
- `emotion_vocabulary` 是否非空，且为字符串数组
- `origin_tag` 是否为受支持的枚举值之一（影视改编/原创剧情），`theme_tags` 是否为数组

校验结果清晰列出每条错误/警告：错误项阻止保存；警告项允许保存但需提示（比如"关键点数量少于3个，可能影响可玩性"这类非阻断性建议）。

## 五、接口层设计（新增，均需鉴权）

- `POST /api/dev/login` `{password}` → 设置session
- `GET /api/dev/scripts` → 列出所有剧本（含状态标记）
- `GET /api/dev/scripts/{id}` → 获取某剧本完整JSON（含内部字段，dev模式下可见）
- `POST /api/dev/scripts/{id}/validate` → 仅做校验，不保存，返回错误/警告列表
- `PUT /api/dev/scripts/{id}` → 保存（内部先触发校验，未通过时返回错误列表且不写入文件）
- `POST /api/dev/scripts/{id}/simulate/start`、`POST /api/dev/scripts/{id}/simulate/message` → 复用 `engine.py` 核心逻辑跑一局测试对局，但响应体中**额外附带**导演与演员的原始JSON输出（正式玩家接口 `/api/session/message` 不返回这些内部字段，仅dev接口返回）
- `GET /api/dev/export` → 打包当前所有 `scripts/*.json`，返回zip供下载

## 六、工程结构变化

```
├── main.py
├── game/
│   ├── ...（原有文件保持不变）
│   ├── script_repository.py   # 新增：统一负责scripts的加载/保存/重载，engine.py和dev模式都通过它读写，不直接碰文件系统
│   └── validator.py            # 新增：静态schema校验逻辑
├── static/
│   ├── index.html              # 玩家端，不变
│   └── dev/
│       ├── index.html          # dev模式页面
│       └── dev.js
```

关键改动：`engine.py` 改为通过 `script_repository.py` 读取剧本，不再直接读文件系统。这样dev模式保存后，下一次玩家请求会读到最新内容，不需要重启进程。

## 七、前端视觉规范（配合另附的两张dev模式示意图：工作台布局、试玩验证面板）

整体原则：**复用玩家端已有的设计语言**（圆角数值、间距、卡片背景、字体层级），而不是另起一套风格；区别只在于信息密度更高、配色更克制、多了调试专用的展示区块。这样整个产品的UI flow观感统一，dev模式看起来像"同一个产品的后台"，而不是另一个粗糙的工具。

### 顶部警示条
- 固定在dev模式页面最顶部，背景用琥珀色系浅底（`#F7E3C2`），文字用对应的深琥珀色（`#6B4A12`），配一个警示图标
- 文案明确提示"改动会立即影响线上玩家端"，让你随时清楚当前操作的影响范围

### 整体布局：列表 + 详情面板
- 左侧固定宽度的剧本列表栏（约220px），每个剧本卡片式展示：标题 + 状态圆点（绿色=已发布，琥珀色=草稿未保存），选中项用浅灰底高亮
- 列表底部并排放置"上传"/"导出全部"两个次要按钮，图标+文字，不需要强调色
- 右侧为详情面板：顶部是剧本名+文件名（灰色小字），右上角三个操作按钮："下载"（次要样式）、"验证"（次要样式）、"保存并发布"（唯一使用强调色`#7F77DD`的按钮，视觉上突出这是关键操作）

### JSON编辑区
- 用等宽字体（`ui-monospace, monospace`）展示/编辑JSON内容，字号12px左右，背景用比卡片底色更深一级的浅灰，圆角8px，与玩家端的卡片圆角保持一致的视觉语言
- 校验结果展示在编辑区上方，用小卡片呈现：通过时用绿色勾选图标+"校验通过"，并列出非阻断性提示；不通过时改用警示配色，逐条列出错误信息

### 试玩验证面板
- 复用玩家端的对话气泡样式（左右布局、圆角14px贴角4px、强调色气泡），保证"测试对局"和"真实对局"视觉一致，减少你在两种模式之间切换时的认知负担
- AI角色消息旁展示情绪标签徽章，与玩家端一致
- **调试信息区块**：每轮对话下方用虚线边框（`0.5px dashed`）的独立区块呈现，与实际对话气泡形成弱化但可辨识的视觉区分；区块内可折叠展开（默认展开即可，demo阶段不强求默认收起），分别用小标题"导演 Agent"/"演员 Agent"展示各自的结构化字段内容（等宽字体，浅色卡片背景），字段名和值可以用简单的 `key: value` 纯文本形式呈现，不需要做语法高亮
- 底部数值条复用玩家端的进度条组件（同样的颜色语义逻辑），只是尺寸更紧凑一些（高度5px而非6px，字号更小），因为dev模式信息密度更高
- 输入框placeholder明确写"以玩家身份输入测试对话…"，提醒你当前是在模拟视角操作

## 八、剧本上传与下载

在原有"导出全部剧本"（打包zip）基础上，补充**单个剧本的上传/下载**能力，方便你在本地编辑好一份剧本JSON后直接导入，或者把某一个剧本单独导出分享/备份：

- **下载单个剧本**：剧本列表每一项旁增加"下载"按钮，导出该剧本的完整JSON文件（文件名建议用 `{script_id}.json`）
- **上传剧本**：提供一个文件选择/拖拽上传入口，上传一个 `.json` 文件后：
  1. 自动跑一遍第四节的静态schema校验，校验不通过则展示错误列表，不导入
  2. 校验通过后，若 `id` 与已有剧本冲突，提示是"覆盖已有剧本"还是"作为新剧本导入（自动重命名id，如追加`_2`后缀）"，由你选择
  3. 导入成功后剧本立即出现在列表中，可以直接编辑或试玩验证
- 上传功能让你可以**在本地用文本编辑器/AI辅助写好剧本草稿后批量导入**，而不必每次都在dev模式的文本框里从头编辑，这对后续内容量增多后的迭代效率很重要

### 接口层新增

- `GET /api/dev/scripts/{id}/export` → 下载单个剧本JSON
- `POST /api/dev/scripts/import` → 上传剧本JSON文件，内部逻辑按上方三步处理，返回校验结果或导入成功的剧本id

## 九、编辑器调试信息深化 + 数值/节奏引擎面板

### 1. 调试信息扩展：原始prompt、响应耗时、token用量

在试玩验证面板已有的"调试信息"区块基础上（已展示导演/演员的结构化JSON输出），为每个agent调用额外记录并展示：

- **原始prompt**：该次调用最终发送给LLM的完整system prompt与user prompt文本（即 `prompt_manager.py` 渲染后的最终结果，而非模板原文），方便直接看到"这一轮实际喂给LLM的是什么"，用于排查prompt效果不符预期的问题
- **响应耗时**：拆分为"首token时间(TTFT)"与"总耗时"两项。为准确测量TTFT，dev模式的模拟调用需要对LLM API使用**流式(streaming)请求**（仅dev的simulate接口内部改为流式并计时，正式玩家端接口不需要改动），从发起请求到收到第一个内容chunk记为TTFT，到收到完整响应记为总耗时
- **Token用量**：记录每次调用的 `input_tokens`/`output_tokens`（从API响应的usage字段读取），导演和演员调用分别展示各自用量

调试信息区块展示结构（在原有JSON基础上扩展）：

```
导演 Agent
  ├─ 原始Prompt（默认折叠，点击展开，等宽字体，system/user分别显示）
  ├─ 原始输出（已有）
  ├─ 首token: 420ms · 总耗时: 1380ms
  └─ Token用量: 输入 812 · 输出 96

演员 Agent
  ├─ 原始Prompt（默认折叠）
  ├─ 原始输出（已有）
  ├─ 首token: 310ms · 总耗时: 890ms
  └─ Token用量: 输入 654 · 输出 42
```

原始prompt文本量较大，默认折叠；耗时与token用量直接以小字常显，不需要折叠。

### 2. 调试记录导出

试玩验证面板新增"导出本局调试记录"按钮：
- 点击后打包当前测试对局全部轮次的调试数据（原始prompt、原始输出、耗时、token用量）为JSON文件下载
- **不做服务端持久化**——数据只存在于当次测试会话内存中，导出即下载，刷新页面或开新对局即丢失，这是刻意的轻量设计
- 文件命名建议含剧本id与时间戳，如 `debug_example_script_20260708_1432.json`

### 3. 数值/节奏引擎面板（轻量版）

目标：**不调用LLM**、纯静态计算，快速判断当前数值/关键点配置在数学上是否合理。

做成编辑器里与"JSON编辑""试玩验证"并列的一个标签页——**"假设路径"计算器**：

- 顶部展示当前草稿的关键参数：各数值初始值、win_condition、lose_condition、max_turns（直接读当前JSON草稿，未保存也能看效果）
- 主体列出全部 `key_points`/`pitfalls`，每项前加复选框，默认不勾选
- 勾选/取消即"假设玩家命中了这些内容"，面板**实时重算**：
  - 各数值的假设最终值（初始值 + 已勾选项`hit_stat_changes`总和，按min/max clamp）
  - 是否达成win_condition / 触发lose_condition / 均未达成，用玩家端同款语义色提示（绿=可胜利，红=会失败）
  - 达成当前假设所需最少轮次（按"一轮最多命中1个关键点"简化估算）与 `max_turns` 的余量提示
- win/lose条件判断逻辑**复用** `validator.py`/`engine.py` 里已有的解析函数，不在这个面板里重新实现一套，避免两处逻辑不一致

这个面板解决的问题是：调整某个关键点的数值权重后，不需要真跑一轮LLM对话，勾选几下就能知道"玩家理想命中这些点，数值曲线够不够触发胜利、轮次够不够"。更复杂的路径模拟（非理想路径、随机踩雷）留到v2.0，本版本只做这个确定性版本。

## 十、验证要求（针对本次改动本身）

1. 用密码进入 `/dev`，确认未授权访问被正确拒绝
2. 编辑一个剧本字段（比如改一句 `teaser`），点击保存，刷新玩家端首页，确认改动**立即生效**，无需重启/重新部署
3. 故意构造几种校验错误（如 `initial` 超出 min/max 范围、`hit_stat_changes` 引用不存在的数值名、`win_condition` 写错语法），确认校验模块能准确报错并阻止保存
4. 在试玩验证面板走一轮完整对局，确认调试信息面板正确展示导演与演员的原始JSON，且与"玩家视角"展示内容逻辑对应
5. 点击"导出全部剧本"，确认下载内容完整，可以直接替换本地 `scripts/` 目录使用
6. 单独下载一个剧本JSON，修改后重新上传，确认能正确触发覆盖/新建选择逻辑，且导入后剧本立即可编辑、可试玩验证
7. 故意上传一份缺字段或格式错误的JSON文件，确认上传流程会先校验、报错，且不会导入到剧本列表中
8. 在试玩验证面板走几轮对局，确认调试信息区块正确展示原始prompt（system/user分别可见）、首token耗时与总耗时、input/output token数，且数值与实际LLM调用情况大致吻合（可以对照API后台或日志核对）
9. 点击"导出本局调试记录"，确认下载的JSON文件包含完整的多轮调试数据，且刷新页面后该数据不再保留（验证未做持久化的预期行为）
10. 在数值/节奏引擎面板中勾选不同组合的关键点/减分点，确认实时计算结果（假设最终数值、胜负状态、轮次余量提示）与手动心算结果一致，并确认其判断逻辑与实际试玩对局中真实触发的胜负结果一致（可以用同一份剧本对照试玩验证面板的实际结果交叉检验）
11. 部署到Render后验证：dev模式改动在当次运行期间对所有访客立即生效；模拟一次重新部署（如推送一次无关代码改动触发redeploy）后，确认之前未导出的改动会丢失——这是预期内的限制而非bug，用来确认你对当前持久化方案边界的认知是准确的
