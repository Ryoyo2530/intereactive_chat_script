# 入戏 v1.4 —— 收尾打磨：轻量生产化、署名与分享、邀请码

## 本版本目标与定位

v1.0～v1.3 已经把核心玩法（导演/演员双agent、关键点系统、分类标签、dev模式验证沙盒、"世界残响"视觉、后端目录重构）跑通并部署稳定。**v1.4 不再动框架和玩法**，目标是把demo从"我自己能跑"打磨到"可以放心分享给朋友和小范围网友玩"，围绕五件事：

1. 轻量生产化（Render免费层下的性能、内存、可观测性）
2. 署名与联系方式（水印）
3. 用户侧下载（图片）与分享
4. 邀请码（控制访问量/成本）
5. 一些低成本的小创意 + 安全细节

**本版本明确不做**：账号体系、数据库/真正的持久化存储、`engine.py` 内部职责拆分、"分享可还原的实时会话链接"（分享=静态快照导出，不是复活一局可继续玩的session）、邀请码的精细颗粒度管理（每人一码+用量统计持久化，这个留到确认真的需要之后再做）。

## 一、现状复核（基于 develop 分支实际代码）

在设计新东西之前，先把当前实现里几个和这次收尾直接相关的事实过一遍：

- **Session 存储**：`game/core/session.py` 用一个进程内 `dict` 保存所有对局状态，**没有任何过期/驱逐机制**。只要进程不重启，session会一直占着内存。Render免费层本身会在15分钟无请求后休眠（休眠=进程重启=内存清零），平时看不出问题，但如果哪天短时间内有较多朋友同时开局又中途弃玩，存活session会一直累积到下次休眠为止，这是本次要补的洞。
- **剧本读写**：`game/content/script_repository.py` 用整目录 `rglob` 扫描 + 内存cache，写入(`save`)是"直接覆盖原文件"的同步写入，**没有原子写保护**——如果写入过程中进程被Render重启（免费层完全可能），文件有被写坏成半截JSON的风险。规模小，这个问题目前没暴露，但顺手修一下成本很低。
- **中间件**：`game/middleware.py` 已经有 `RateLimitMiddleware`（按IP，`/api/*` 默认120次/分钟）和 `StaticCacheMiddleware`（给`/echoes/` `/vendor/` 等静态资源加缓存头），说明v1.3的后端优化已经落地了一部分，这次是查缺补漏而不是从零开始。
- **LLM Key 来源**：`game/llm/config.py` 是 **BYOK优先、服务端env key兜底** 的设计——玩家在"模型配置"tab里没填自己的key时，会**静默使用你在Render环境变量里配的 `LLM_API_KEY`**。这是你"怕分享出去人太多"这个担忧的真实技术根源：不是访问量本身有风险，是**没主动配置key的访客默认在花你的钱**。这个事实直接决定了邀请码该设计成什么样（见第五节）。
- **访问控制**：目前只有 `/dev` 有密码门（`DEV_MODE_PASSWORD`），玩家端首页和所有 `/api/session/*` `/api/scripts` 接口是**完全公开**的，任何拿到URL的人都能直接玩，也就都能消耗你的env key配额。
- **Key脱敏**：`mask_api_key` / `public_dict` 已经做了debug导出时不泄露完整key，这点做得不错，本次不需要动。

以上是接下来所有设计的事实基础，避免凭空加机制。

## 二、性能与轻量生产化

### 1. Session 内存治理

不引入Redis/数据库，纯内存方案继续，但加三层保护：

- **TTL惰性清理**：`create_session` 时给每个session记录 `last_active_at`；每次 `get_session`/`update_session` 时更新它。新增一个轻量的清理入口——不需要后台定时任务（Render免费层不建议常驻后台线程消耗仅有的CPU配额），采用**惰性清理**：每次 `create_session` 被调用时，顺手扫一遍并剔除超过例如30分钟无活动的session。剧本对局本身很少会跨越30分钟不操作，这个阈值不会误伤正常玩家。
- **并发上限**：加一个简单的"当前存活session数"计数，超过一个上限（比如200，按Render免费实例内存粗估留足余量）时，`start_session` 直接返回一个友好提示"当前体验人数较多，请稍后再试"，而不是无限制accept导致OOM。这个数字可以做成配置项，方便你未来根据实际内存表现调整。
- **单session体积上限**：`history` 数组理论上会随对话轮次无限增长（虽然剧本都配了`max_turns`，但如果玩家在`game_over=true`之后又调了一次message接口，或者某个剧本`max_turns`配得很大），建议在 `engine.py` 层面加一个硬性轮次上限保护（比如无论剧本配置多少，全局硬顶50轮），防止极端输入把单个session的内存/token成本推高。

### 2. 网络与容错

- 确认 `game/llm/client.py` 里对外部LLM请求设置了**合理的超时**（比如30-45秒）并且只做**一次轻量重试**（避免网络抖动时无限重试推高你的API账单和响应时间）。如果目前没有超时设置，这是本次优先修的项，避免一个卡住的外部请求占住Render免费层本就紧张的worker资源。
- 失败时给玩家的提示要走"角色语塞"式的兜底文案（`director_system`/`roleplay`已经有解析失败兜底），网络层失败也应该复用同一套"游戏内语言"的提示，而不是暴露`Connection timeout`这类原始报错。

### 3. 响应体积

- 建议给 FastAPI 应用加 `GZipMiddleware`（Starlette自带，几行代码），目前 `middleware.py` 里还没有这个，对JSON响应和静态HTML/JS/CSS都有直接的传输体积收益，Render免费层出口带宽也有限，这个性价比很高。
- Tailwind CDN 版本在生产环境是运行时JIT编译，有一定首屏开销，但考虑到当前规模（几个朋友体验），换成构建版本收益不大、还要多引入构建步骤，**本次不做**，先记在后续技术债里。

### 4. 冷启动体验（而不是硬解决冷启动本身）

Render免费层15分钟无请求会休眠，之后第一个请求要等实例重新拉起（通常30-60秒）。**不建议**做定时ping保活（一是违背免费层的使用初衷，二是如果流量变多，保活反而会让实例几乎常驻、更容易触碰免费层的月度小时数上限）。更划算的做法是在前端把这段等待**体验化**：首次进入选择页如果`/api/scripts`迟迟没有响应，展示一句和"世界残响"美术调性一致的过渡文案（比如"正在唤醒沉睡的剧本…"），而不是让朋友看着一个空白页以为挂了。这个点在第六节的"小创意"里也会呼应。

### 5. 轻量可观测性（不引入外部服务）

不需要接Sentry/Datadog这类东西，纯内存计数器 + 现有的Render日志面板就够用：

- 新增一个仅 dev 模式可见的 `GET /api/dev/stats`，返回：当前存活session数、今日累计开局次数、今日累计LLM调用失败次数、平均响应耗时（可以用简单的滑动窗口或者干脆是"进程启动以来"的累计值，重启清零，符合你"不做持久化"的一贯取舍）。这样你不用去翻Render的日志流就能一眼看到"今天有多少人在玩、稳不稳"。
- 复核 `game/logging_config.py`：确认关键节点都有日志——LLM调用的成功/失败/耗时、rate limit触发（已有）、session创建/清理、剧本保存。日志里**不要**打印玩家的完整对话内容（隐私考虑，朋友间分享更要注意这点），可以只记录长度/轮次这类元信息。

## 三、署名与联系方式（水印）

目的：分享出去后，别人如果转发截图/图片，能带着你的署名走，同时也顺手放一个免责声明保护自己。

### 展示位置（三处，风格统一，不做成弹窗）

1. **选择页页脚**：一行小字，字号11px、灰色次要文字色（延续v1.3"世界残响"色板 `#232323`正文/`--echo-border`边框的次要文字色阶），内容类似"入戏 · by [你的名字/昵称] · [联系方式，如小红书/微信号/邮箱任选]"。可以和第六节的"作者的话"折叠区块共用一个入口，不重复占版面。
2. **结局结算卡片底部**：在"再玩一次/换一个剧本"按钮下方加一行更小的署名+一句免责声明，比如"内容由AI生成，仅供娱乐体验" —— 这句话尤其对"影视同人"类剧本重要：这些是基于影视角色的**同人创作**，加一句"非原作品官方内容，人物为同人演绎"的声明，分享给不熟悉背景的陌生网友时能减少误解，也是对自己的一层保护。
3. **分享图片本身**：见第四节，图片角落固定打印署名，这是"防止被裁掉来源到处传"的实际手段，比页面上的文字署名更重要——页面署名只有点进来的人能看到，图片署名是跟着内容走的。

### 风格要求

延续既有设计token，不要另起一套视觉语言：Georgia + Noto Serif SC，字间距`0.03em`，颜色用正文色的低透明度或专门的次要文字灰，不用醒目的强调色抢视觉——署名是"不经意地留在那里"，不是广告位。

## 四、用户侧下载（图片）与分享

### 设计前提

明确一件事：**这是快照导出，不是可继续的会话分享**。没有账号和持久化的情况下，"分享"能做到的最好效果是"把这一局发生的事变成一张图/一段文字，发给朋友看"，而不是"把链接发给朋友，对方能接着玩你这局"。后者需要持久化存储，这次不做，需要在UI文案上说清楚，避免朋友点开分享链接后困惑。

### 两种导出形式

**1. 精华卡片（默认，主推的"分享"形态）**

内容：剧本标题/章节名 + 结局标题 + 结局文案（2-3句）+ 最终数值回顾 + 1-2条"世界残响"卡片摘录（挑数值变化幅度最大的一两条，而不是全部）+ 角落署名水印。视觉上就是把现有的结局结算卡片直接渲染成图片，天然复用已有的设计和数据，不需要额外设计一套新版式。

**2. 完整回顾（次要选项，"下载完整记录"）**

内容：从头到尾的阅读流全文（角色台词+情绪舞台提示+玩家发言+每次"世界残响"卡片），生成一张长图。这个可能会很长，明确告诉用户"这会生成一张较长的图片"，避免意外。

两者都保留一个"另存为纯文本"的兜底选项（导出`.txt`，把对话原文按顺序列出），成本几乎为零（前端直接拼字符串触发下载），给不需要图片、只想存个文字记录的人用。

### 实现方式

- **纯前端实现，不新增后端持久化接口**。用 `html2canvas`（CDN引入，体积小）把渲染好的DOM区域截图成图片，这样能直接复用现有CSS，不用维护第二套canvas绘制逻辑。需要注意：截图前要等自定义字体（Noto Serif SC）真正加载完成（`document.fonts.ready`），否则截图里可能出现字体闪一下没生效的情况。
- **分享**：移动端优先用 `navigator.share`（如果浏览器支持，且能传图片文件）调起系统分享面板，直接分享到微信/小红书等；不支持的浏览器（多数桌面端）降级为"下载图片"按钮 + 一段可复制的分享文案（剧本标题+结局一句话概括，不含真实链接以外的内容，因为没有可分享的"续局链接"）。
- 唯一可能涉及后端的点：如果希望水印文案/联系方式改起来不用重新发版前端文件，可以做一个小的配置接口读取，但考虑到署名信息几乎不会频繁变，**建议直接硬编码在前端**，减少不必要的接口维护成本，这也符合当前"能少动后端就少动"的取舍。

## 五、邀请码机制

### 先想清楚要挡什么

结合第一节的现状复核：真正的风险点不是"人多"本身，是"没配自己key的人在花你的env key配额"。邀请码要解决的核心问题是**控制陌生流量**，而不是要做一套精细的用户管理系统。基于这个判断，给三个方案按实现成本排序：

| 方案 | 描述 | 成本 | 适合场景 |
|---|---|---|---|
| A. 全站单一暗号 | 复用`/dev`已有的"密码→session cookie"模式，选择页前加一道门，全体朋友共用一个邀请码 | 低（照抄`game/dev/auth.py`的模式） | 熟人小范围分享，够用 |
| B. 每人一码+限次 | 预生成一批邀请码（env变量逗号分隔或单独json文件），每码限定使用次数，用完失效 | 中 | 想知道具体是谁在用、控制单人上限 |
| C. 不做强验证，只做软摩擦 | 页面上明确写"仅限受邀访问"+一个统一暗号，主要挡爬虫/搜索引擎/误闯的陌生人 | 最低 | 纯粹为了不被搜索引擎收录/随缘路过的人点进来 |

**推荐本版本做方案A**，理由：
- 你的担忧是"分享给朋友、小范围网友"场景下的量，不是要做精细的邀请码分发和用量分析——那是B方案要解决的问题，目前没到需要的程度
- 复用现有`/dev`鉴权的模式，代码几乎是抄一遍改个变量名，不引入新的复杂度
- 比C多一层"真正拦下没有码的人"的效果，而C基本挡不住任何认真想进来的人

**具体设计**：
- 新增环境变量 `INVITE_CODE`
- 新增 `game/access.py`：`verify_invite_code(code) -> bool` + 生成一个长期（比如30天）的session token 存 cookie，逻辑和`game/dev/auth.py`平行但完全独立（不要和dev鉴权共用同一个token/cookie名，两者权限含义不同，混用容易出安全隐患）
- 新增 `routers/access.py`：`POST /api/access/verify {code}`
- 前端在选择页最外层加一层轻量拦截：没有有效邀请码cookie时，展示一个和整体视觉一致的邀请码输入页（不是浏览器原生弹窗），通过后正常进入选择页，此后同一浏览器不用重复输入
- **只挡 `/api/session/*` `/api/scripts` 这些真正会触发LLM调用/消耗你配额的接口**，静态资源本身不需要挡（没有实际成本）

### 说清楚边界

邀请码挡得住"随手点进链接的陌生人"，挡不住"朋友把邀请码转给了他的朋友"——这是分享场景下无解的问题，技术上做不到防止已获得邀请码的人二次转发。如果真出现流量失控，更现实的做法是：
1. 换码（改一下env变量重新部署，旧码立即失效，成本极低）
2. 直接跟朋友说一声"先别转发出圈"

不需要为了防"熟人的熟人"这种低概率场景过度设计成B方案的每人限次系统，那个投入产出比不划算，等真的遇到问题再升级。

## 六、小创意 + 安全细节

### 安全/隐私相关

- **`robots.txt`**：新增，`Disallow: /`，避免demo被搜索引擎收录——这是比邀请码更前置的一层保护，防止链接被搜索引擎抓取后随缘出现在陌生人的搜索结果里
- **`noindex` meta**：站点所有页面加 `<meta name="robots" content="noindex, nofollow">`，双重保险（部分爬虫不读robots.txt）
- **日志不落地对话原文**：第二节已提到，这里再强调一次——朋友分享给你玩的对局，对话内容本身可能带一些私人化的表达方式，日志里只记结构化元信息（轮次数、耗时、是否报错），不要整段存对话正文
- **原子写保护**：`script_repository.save()` 改成"先写临时文件，成功后 `os.replace` 原子替换"，防止写入过程中进程被重启导致文件损坏——这个和邀请码/署名关系不大，但顺手修，属于"不容易被注意到但一旦出问题很麻烦"的类型

### 小创意（低成本、体验向）

- **"作者的话"折叠区块**：选择页里加一个默认收起的小区块（不是弹窗，不打断进入体验），点开是一小段你写的话——demo性质说明、AI生成免责声明、联系方式/反馈入口，一次性把第三节的署名和这里的说明合并展示，不用在页面上到处放文字
- **反馈入口**：页脚一个不起眼的"觉得哪里怪？告诉我"链接，指向你的联系方式或一个免费的腾讯问卷/Google表单，低成本收集朋友的真实体验反馈，不需要自建评论区功能
- **冷启动等待文案**：呼应第二节第4点，用"世界残响"调性的过渡文案代替干巴巴的loading，把Render免费层的慢启动"演出化"，而不是让人以为挂了
- **对局"编号"小彩蛋**：结局卡片上加一行类似"第 [剧本ID简写]-[本地时间戳后6位] 场"的小字，纯前端生成，不需要任何持久化，营造"这是独一无二一次演出"的仪式感，也顺带让分享出去的图片天然带一点"这不是模板截图，是真实跑出来的"的可信度

8. **冷启动体验**：故意等实例休眠后再访问，确认过渡文案正确展示而不是白屏
